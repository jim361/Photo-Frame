# 01. 프레임 렌더링 엔진

## 목적

필름 스캔본 위에 프레임과 촬영 정보를 합성해 "인스타에 바로 올릴 수 있는 한 장"을 만든다. 사용자는 스타일 하나를 고르고 정보를 채우면 되고, 레이아웃·타이포·비율은 도구가 책임진다.

## 현재 동작 (v3)

`render(img, o): canvas` — 원본 크기 기준으로 새 캔버스에 합성한다. 미리보기와 내보내기가 같은 함수를 공유한다.

**공통 텍스트 규칙**
- `meta(o)`: 바디·렌즈를 `  ·  `로 연결. 빈 항목은 자동 생략 (`filter(Boolean)`)
- `frameLabel(o)`: 시작 프레임 + 사진 순번, 2자리 패딩 + `A` (예 `07A`) — 필름 엣지 표기 관례
- `trackedText`: `ctx.letterSpacing` 지원 시 사용, 미지원 브라우저는 글자 단위 수동 렌더 폴백

**스타일 4종** (수치 근거는 [DESIGN.md](../DESIGN.md))

| 스타일 | 캔버스 크기 | 내용 |
|---|---|---|
| `film` | W × (H + 0.088W + 0.154W) | 위아래 검정 바 + 퍼포레이션 2줄, 아래 바에 프레임 번호(주황, 좌) + 장비·필름(회색, 우, 대문자) |
| `instax` | 1.11W × (H + 0.055W + 0.27W) | 아이보리 카드, 사진 둘레 옅은 스트로크, 하단 여백에 장비(중앙)/필름(중앙)/프레임 번호(우하단) |
| `bottom` | W × (H + 0.078W) | 사진 아래 여백 띠 — 장비·필름(좌) + 프레임 번호(주황, 우). 밝은/어두운 톤 |
| `top` | W × (H + 0.078W) | `bottom`과 동일하되 띠가 위 |

`downscale(cv, longEdge)`: 내보내기 시 긴 변 기준 축소 (2048/1440/원본), `imageSmoothingQuality: 'high'`.

## 설계 결정

- **원본 해상도로 합성 후 마지막에 축소** — 텍스트·퍼포레이션이 축소 과정에서 자연스럽게 안티앨리어싱된다. 처음부터 작게 그리면 얇은 스트로크가 뭉개진다.
- **모든 수치를 사진 폭 W의 비율로** — 어떤 해상도의 스캔이 와도 프레임 비율이 동일하다.
- **DOM 캔버스 사용 (OffscreenCanvas 아님)** — v3 시점에는 워커 분리 이득보다 단순함을 택했다. 2단계에서 순수 함수로 분리할 때 재검토.
- SVG/CSS 합성이 아닌 Canvas인 이유: 결과물이 픽셀 이미지 파일이어야 하고, 미리보기=결과물 일치가 필요하다.

## 알려진 한계

- **ICC 프로파일 무시** — Adobe RGB 등 임베디드 프로파일이 있는 스캔은 색이 틀어질 수 있다. sRGB 변환 후 사용 권장 (ARCHITECTURE.md 참고).
- **EXIF 회전** — `Image` 요소 경유 로드라 최신 브라우저는 대부분 자동 보정하지만, 구형 경로/일부 환경에서 세로 사진이 눕는다. → 1단계에서 `createImageBitmap(file, { imageOrientation: 'from-image' })`로 명시 처리 예정
- **캔버스 크기 한계 미검사** — 고해상도 스캔 + 프레임이 브라우저 한계(모바일 Safari ~4096²)를 넘으면 조용히 빈 결과가 나온다. → 1단계에서 자동 축소 + 안내 예정
- 표시 항목을 개별 토글할 수 없다 (빈 칸으로 비우는 것만 가능) → [04. 촬영 정보](04-shot-info.md)의 표시 설정으로 해결 예정

## 개선 계획
