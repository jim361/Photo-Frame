# 09. 프로젝트 저장·복원

## 목적

여러 장의 순서와 장별 촬영 정보 수정을 끝내기 전에 브라우저를 닫아도 작업을 이어갈 수 있게 한다. 사진 데이터를 localStorage나 프로젝트 파일에 복제하지 않고, 사용자가 보관한 원본과 다시 연결하는 로컬 전용 흐름이다.

## 현재 동작

- 우측 **프로젝트** 영역에서 `프로젝트 저장` / `프로젝트 열기`를 제공한다.
- 저장 파일은 `.photoframe.json`이며 다음 정보를 포함한다.
  - 스트립의 사진 순서와 현재 선택 장
  - 원본 식별 정보(파일명·크기·마지막 수정 시각·MIME 형식)
  - 전역 촬영 정보, 표시·방향별 레이아웃, 스타일, 내보내기 설정
  - JPEG에서 읽은 장별 EXIF와 사용자가 수정한 장별 값
  - 현재 적용 중인 로고 1개(400px 이하로 보관된 기존 팔레트 dataURL)
- **사진 원본 바이트는 포함하지 않는다.** 프로젝트를 열면 설정을 먼저 복원하고 사진이 비어 있는 상태에서 원본 선택을 안내한다.
- 원본을 다시 불러오면 파일명·크기·마지막 수정 시각으로 우선 연결하고, 일치하지 않으면 같은 파일명을 폴백으로 쓴다. 연결된 사진은 프로젝트 순서로 자동 정렬된다.
- 일부 사진만 선택했으면 `연결된 장수/전체 장수`를 표시하고 나머지를 추가로 불러올 수 있다.
- 프로젝트의 로고는 기존 팔레트를 지우지 않고 병합한다. 같은 이름의 다른 이미지가 있으면 `(프로젝트)` 이름으로 별도 저장한다.
- 프로젝트를 여는 시점에 사진이 이미 있으면 현재 사진을 비울지 확인한다.
- 파일 제목이 있으면 `제목.photoframe.json`, 없으면 `PhotoFrame-YYYY-MM-DD.photoframe.json`으로 저장한다. 저장 폴더가 연결돼 있으면 그 폴더를 쓰고, 아니면 브라우저 다운로드로 저장한다.

## 설계 결정

- **사진 비포함 manifest** — 수십 장의 원본을 JSON/dataURL로 복제하면 파일이 지나치게 커지고 메모리 사용이 늘어난다. 프로젝트는 작업 상태만 보관하고 원본은 사용자가 다시 선택한다.
- **개인 팔레트와 프로젝트 분리** — 장비·레이아웃 프리셋 전체는 브라우저의 개인 설정이므로 프로젝트로 덮어쓰지 않는다. 결과 재현에 필요한 현재 설정과 활성 로고만 포함한다.
- **버전이 있는 형식** — `kind: photoframe-project`, `version: 1`을 검사해 다른 JSON이나 향후 호환되지 않는 버전을 잘못 적용하지 않는다.

## 알려진 한계

- 원본 파일명이 바뀌면 자동 연결할 수 없다. 같은 이름만 남아 있으면 파일명 폴백으로 연결하지만 다른 사진을 잘못 고를 가능성이 있다.
- 프로젝트 자체에는 사진이 없으므로 원본 파일을 함께 보관해야 한다.
- 현재 장별 저장 범위는 구현된 장별 모델과 동일하게 카메라·렌즈·날짜·설정값이다. 필름·기타/메모/표시 토글의 장별화는 [07. 장별 설정 오버라이드](07-per-photo-override.md) 구현과 함께 확장한다.

## 수용 기준

- [x] 사진 원본이 프로젝트 JSON에 포함되지 않는다
- [x] 사진 순서·현재 설정·장별 EXIF 수정값이 저장된다
- [x] 프로젝트를 연 뒤 원본을 순서와 무관하게 선택해도 저장된 순서로 복원된다
- [x] 일부 원본만 선택한 뒤 나머지를 추가 연결할 수 있다
- [x] 잘못된 JSON을 열어도 현재 작업을 지우거나 초기화를 중단하지 않는다
- [x] 기존 개인 로고 팔레트를 프로젝트가 덮어쓰지 않는다
