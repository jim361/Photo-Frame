# PhotoFrame 경쟁 제품 분석

> 조사 기준일: 2026-07-20
>
> 대상: 사진에 프레임과 EXIF·촬영 정보를 합성하는 도구. 일반 사진 편집기와 단순 콜라주 앱은 직접 경쟁군에서 제외했다.

## 결론

PhotoFrame은 **데스크톱 묶음 작업·직접 배치·사진 원본을 포함하지 않는 프로젝트 저장과 재연결·프라이버시**에서는 직접 경쟁 제품과 비교해도 상위권이다. 단순 실험작보다는 실제 사용 가능한 v1 제품에 가깝다.

반면 **모바일 완성도·프레임 종류·입력 포맷·다국어·첫인상과 공개 사용 지표**는 Paddie, OneLine, Liit 같은 상용 앱보다 뒤처진다. 현재 부족한 것은 합성 엔진보다 제품을 처음 접하고 반복 사용하는 경험이다.

따라서 PhotoFrame은 범용 사진 편집기를 따라가기보다 **“사진을 외부로 보내지 않고, 서로 다른 디지털·필름 사진 한 묶음을 같은 규칙으로 정리하는 로컬 프레임 도구”**를 목표로 발전시키는 편이 유리하다. 현재 장별 정보는 일부 필드만 분리되므로 이 문장은 완전한 장별 오버라이드 이후의 목표 포지셔닝이다.

## 판단 방법과 한계

- 기능은 각 제품의 공식 저장소, 공식 웹사이트, App Store·Google Play 설명을 기준으로 확인했다.
- 채택 정도는 다운로드, 평가 수, GitHub 스타처럼 공개된 신호만 사용했다. 다운로드 수와 스타 수는 활성 사용자 수와 같지 않으며, 수치가 없다고 사용자가 없다는 뜻도 아니다.
- 개인정보 표시는 공개 소스에서 확인되는 코드와 스토어의 개발자 신고를 구분했다. 비공개 앱의 “데이터 수집 없음” 표시는 PhotoFrame처럼 코드를 직접 검증할 수 있다는 뜻은 아니다.
- 결과물의 미감과 픽셀 품질은 동일 원본·동일 출력 조건의 블라인드 테스트를 하지 않았으므로 순위를 매기지 않았다. 대신 재현 가능한 기능과 작업 흐름을 비교했다.
- 이 문서에서 **Paddie Legacy**와 **New Paddie**는 같은 개발자 계보의 별도 제품이다. **EXIF Frame**은 [jeonghyeon-net/exif-frame](https://github.com/jeonghyeon-net/exif-frame)과 그 공식 웹·모바일 배포판을 뜻한다.

| 제품 소개·채택 증거 | 약함 | README 데모 이미지와 영문 소개가 없고, 공개 사용자 지표도 아직 없다. 기능에 비해 첫인상과 발견 가능성이 약하다. |
| 유지보수성 | 혼합 | 의존성 0과 단일 배포물은 강점이지만, 3천 줄이 넘는 단일 HTML과 자동 회귀 검사가 없는 점은 장기 비용이다. |

### Paddie와 EXIF Frame을 따로 볼 때

- **Paddie Legacy 대비:** PhotoFrame은 설치·의존성·프로젝트 이동·배치 레이아웃 조작에서 낫다. Paddie는 포맷 변환과 EXIF 토큰의 폭이 넓다.
- **New Paddie 대비:** PhotoFrame은 앱 설치 없는 실행, 공개 코드를 통한 프라이버시 검증과 데스크톱 롤 작업이 낫고, Paddie는 모바일·스타일·다국어·콜라주가 낫다. PhotoFrame은 이동 가능한 프로젝트 파일을 제공하며, Paddie의 공개 설명에서는 같은 형태의 프로젝트 내보내기를 확인하지 못했다. 두 제품은 “집중된 로컬 도구”와 “종합 모바일 편집기”로 방향이 다르다.
- **EXIF Frame 대비:** PhotoFrame은 사용자 정의 깊이와 프로젝트 복원이 낫고, EXIF Frame은 테마 선택 속도·웹판 ZIP·다국어·공개 채택 신호가 낫다.

## PhotoFrame의 확실한 차별점

1. **외부 전송 0이 아니라 외부 요청 0:** 사진만 로컬이라는 서비스는 있어도 analytics·폰트·업데이트 확인까지 없는 제품은 드물다.
2. **한 묶음 안의 서로 다른 촬영 정보:** 디지털 카메라, 폰, 필름 스캔을 함께 올릴 때 전역 기본값과 장별 정보가 공존하는 방향은 실사용 가치가 높다.
3. **프리셋 선택보다 깊은 직접 조작:** 요소별 이동, 스냅, 프레임 크기 핸들, 방향별 프로파일, 50단계 돌아가기는 결과물을 직접 설계하는 도구에 가깝다.
4. **사진을 넣지 않는 프로젝트 파일:** 원본 노출과 파일 크기를 줄이면서 순서·EXIF 수정·레이아웃·활성 로고 1개·현재 사용 중인 파일 글꼴 최대 2개를 저장한다. 다시 미리 보거나 내보낼 때는 보관한 원본 사진을 재선택해야 한다.
5. **로컬 타이포그래피:** 외부 폰트 서버 없이 영문·한글 글꼴을 따로 선택하고, 사용자가 고른 파일 글꼴만 프로젝트에 포함한다.
6. **미리보기와 내보내기의 같은 렌더 경로:** 화면과 최종 결과가 달라지는 위험을 구조적으로 줄인다.

### 권장 실행 순서

아래는 **제품 가치 기준 우선순위**이지 그대로 착수하는 구현 순서는 아니다. 완전한 장별 오버라이드는 [기능 명세](features/07-per-photo-override.md)의 데이터·렌더 의존성을 먼저 정리하고 UI는 구현 전 다시 확인한다. 영어 UI는 현행 “UI 언어는 한국어” 규약과 [국제화 명세](features/08-i18n.md)의 마이그레이션 순서를 변경하기 전 사용자 확인이 필요하며, README 영문 요약은 그와 별개로 먼저 추가할 수 있다.

1. 완전한 장별 오버라이드
2. 내보내기 충돌 방지·진행 표시
3. README 결과 예시와 영문 요약
4. 배치 선택·적용 상태 표시
5. 프리셋 JSON 이동
6. 부분 EXIF·접근성·다국어 기반

## 의도적으로 따라가지 않을 것

- 필터, 색보정, 크롭, 스티커, 그리기: 프레임 합성기라는 범위를 흐린다.
- 내장 카메라 브랜드 로고 묶음: 상표 사용과 배포 책임이 커진다. 사용자가 권리를 확인한 로고만 직접 불러오는 현재 방식이 안전하다.
- 계정, 클라우드 동기화, analytics, 템플릿 마켓: 외부 요청 0과 단일 HTML의 장점을 잃는다.
- “가장 많은 프레임” 경쟁: 네 가지 스타일의 방향별 완성도와 혼합 장비 배치가 더 강한 차별점이다.
- 외부 변환 서버나 대형 디코더를 통한 HEIC·RAW 지원: 절대 규칙과 유지보수 비용을 훼손한다.

## 제품 포지셔닝 제안

> **PhotoFrame — 사진을 올리지 않고, 서로 다른 디지털·필름 사진 한 묶음을 같은 규칙으로 완성하는 로컬 프레임 도구**

경쟁 제품보다 기능 수가 많다고 말하기보다는 다음 세 문장을 반복하는 편이 정확하다.

- 사진과 촬영 정보는 브라우저 밖으로 나가지 않는다.
- 한 장이 아니라 한 묶음을 같은 스타일로 정리한다.
- 완전한 장별 오버라이드가 구현되면 기본값은 함께 쓰고, 다른 정보만 사진별로 바꾼다.
