# FilmFrame — Claude Code 인수인계 프롬프트

> 사용법: 새 프로젝트 폴더에 `filmframe.html`(현재 v3 단일 파일)을 넣고,
> 그 폴더에서 Claude Code를 실행한 뒤 아래 내용 전체를 첫 프롬프트로 붙여넣는다.

---

## 프로젝트 컨텍스트

**FilmFrame**은 필름 스캔본에 프레임과 촬영 정보(바디/렌즈/필름)를 얹어 인스타그램 업로드용 이미지를 만드는 **로컬 웹 도구**다. 서버 없음, 계정 없음, 사진이 브라우저 밖으로 나가지 않는 것이 핵심 원칙이다.

**만든 동기 (README와 CONCEPT에 반영할 것):**
- 제작자는 Nikon F5로 필름 풍경 사진을 찍고, 인스타그램에 롤 단위 회고 형식으로 올린다.
- 현상소에서 스캔본이 돌아올 때쯤이면 어떤 컷을 어떤 렌즈로 찍었는지 기억나지 않는 경우가 많다. 그래서 **불확실한 정보는 표시하지 않을 수 있는 것**(항목별 표시 토글)이 이 도구의 차별점이다.
- 기존 대안(라이트룸 프리셋, 포토샵 액션, 유료 앱)은 매 장 텍스트를 수정하기 번거롭거나, 사진을 서버에 올려야 한다.

**주 사용자:** 필름 사진을 인스타에 올리는 사람. 롤 하나(5~36장)를 한 번에 처리하고 싶어 한다.

**현재 상태:** 프로토타입 v3가 단일 HTML 파일(`filmframe.html`, 약 900줄, 바닐라 JS + Canvas 2D)로 동작한다. 이 파일이 유일한 소스이며, 기능 명세의 원본(source of truth)이다. 문서 작성 전에 반드시 전체를 읽어라.

---

## 현재 구현된 기능 (v3 기준)

1. **프레임 스타일 4종** — Canvas 2D로 합성
   - `film` 필름 스트립: 위아래 퍼포레이션(스프로킷 홀), 아래 바(위 바의 1.75배 높이)에 엣지 프린트 스타일 텍스트. 필름명은 주황(#e8873a) 각인, 장비는 회색
   - `instax` 인스탁스: 아이보리(#f6f3ec) 카드, 하단 여백 = 사진 폭의 27% (실제 instax mini 비율에서 유도)
   - `bottom` / `top` 여백: 띠 높이 = 사진 폭의 7.8%(한 줄) / 11.2%(두 줄), 밝은/어두운 톤 선택
2. **줌/팬 뷰포트** — transform 기반. 휠 줌(커서 고정점), 드래그 팬, 툴바(−/+/맞춤/1:1), 키보드(+, −, 0, 1), ResizeObserver로 맞춤 모드 유지. 사진 로드 시 화면 맞춤이 기본
3. **필름 스트립(하단 썸네일 바)** — 클릭 선택, HTML5 DnD로 순서 변경(드롭 위치 주황 인디케이터), × 삭제
4. **촬영 정보 + 표시 설정** — 바디/렌즈/필름 입력(datalist 자동완성), 접이식 "표시 설정" 패널: 항목별 표시 체크박스 3개 + 한 줄/두 줄 레이아웃. 두 줄이면 윗줄 바디·렌즈, 아랫줄 필름. localStorage에 자동 저장
5. **프리셋** — 이름/장비/스타일/표시설정을 localStorage에 저장, 칩 UI로 적용·삭제. localStorage 접근 실패 시 메모리 폴백
6. **내보내기** — File System Access API(`showDirectoryPicker`)로 폴더 지정 저장, 미지원 환경은 다운로드 폴백(장당 350ms 딜레이). 긴 변 2048/1440/원본 리사이즈. PNG 고정. 파일명 `원본이름_framed.png`
7. **UI 언어는 한국어**, 다크 톤(암실/라이트테이블 컨셉), 우측 패널 왼쪽 가장자리에 퍼포레이션 장식

## 알려진 결함 (문서화 + 수정 대상)

- **색 프로파일**: Canvas가 임베디드 ICC 프로파일(Adobe RGB 등)을 무시해 색이 틀어질 수 있음. `createImageBitmap` + `colorSpaceConversion` 검토. 완전 해결이 어려우면 한계로 명시
- **EXIF 회전 미처리**: 세로 사진이 눕는다. `createImageBitmap(file, { imageOrientation: 'from-image' })`로 해결
- **캔버스 크기 한계**: 고해상도 스캔 + 프레임이 모바일 Safari 한계(~4096²)를 넘으면 조용히 실패. 크기 검사와 사용자 안내 필요
- **메모리**: 원본 Image 객체를 전부 유지. 썸네일은 축소본으로 분리해야 함
- **모바일 터치**: 핀치 줌 없음(pointer 이벤트 기반이라 확장 가능)

## 너(Claude Code)의 작업 지시

### 0단계 — 저장소 및 문서 골격 (가장 먼저, 코드 수정 전에)

git 저장소를 초기화하고 아래 문서 구조를 **먼저** 만든다. 각 문서는 `filmframe.html`을 실제로 읽고 분석해서 작성한다. 추측으로 채우지 마라.

```
filmframe/
├─ README.md                      # 마지막에 작성 (아래 참고)
├─ LICENSE                        # MIT
├─ docs/
│  ├─ CONCEPT.md                  # 아이디어 컨셉
│  ├─ DESIGN.md                   # 디자인 시스템
│  ├─ ARCHITECTURE.md             # 구조와 마이그레이션 계획
│  └─ features/
│     ├─ 01-frame-rendering.md    # 프레임 렌더링 엔진
│     ├─ 02-viewport.md           # 줌/팬 뷰포트
│     ├─ 03-filmstrip.md          # 썸네일 스트립 (선택/정렬/삭제)
│     ├─ 04-shot-info.md          # 촬영 정보 + 표시 설정
│     ├─ 05-presets.md            # 프리셋 저장/적용
│     ├─ 06-export.md             # 내보내기 + 파일 시스템
│     ├─ 07-per-photo-override.md # [미구현] 장별 설정 오버라이드
│     └─ 08-i18n.md               # [미구현] ko/en
└─ legacy/
   └─ filmframe.html              # v3 원본 보존 (참조용, 수정 금지)
```

**CONCEPT.md에 들어갈 것:** 문제 정의(현상 후 렌즈 기억 문제 포함), 대상 사용자, 핵심 원칙 3가지(① 사진은 브라우저 밖으로 나가지 않는다 ② 불확실한 정보는 표시하지 않을 수 있다 ③ 롤 단위 일괄 처리가 기본 흐름), 기존 대안과의 비교, 스코프에서 제외하는 것(색보정·크롭 등 편집 기능은 안 만든다 — 이 도구는 프레임만 얹는다).

**DESIGN.md에 들어갈 것:** 디자인 컨셉("암실 라이트테이블"), 컬러 토큰 전체(`--base #14130f`, `--panel #1c1a15`, `--panel-hi #24211b`, `--rebate #6e6a5f`, `--bone #efeae0`, `--edge #e8873a`, `--line #322e26`), 타이포(UI는 모노스페이스가 라벨/데이터 담당, 시스템 산세리프가 본문), 시그니처 요소(우측 패널의 퍼포레이션 장식), 캔버스 렌더링 디자인 수치(퍼포레이션 크기·간격 공식, 각 스타일의 여백 비율과 그 근거 — instax 27%는 실제 instax mini 카드 비율에서 온 것), 상호작용 원칙(로드는 항상 화면 맞춤, 타이핑 중 줌 유지, 스타일 변경 시에만 줌 리셋).

**각 feature md의 공통 형식:**
```markdown
# NN. 기능명
## 목적          — 왜 존재하는가 (사용자 관점 한 단락)
## 현재 동작      — v3 기준 상세 스펙 (코드 분석 기반)
## 설계 결정      — 왜 이렇게 만들었는가, 검토했다 버린 대안
## 알려진 한계    — 결함, 엣지 케이스
## 개선 계획      — 단계별 (아래 로드맵과 연결)
## 수용 기준      — 완료 판정 가능한 체크리스트
```

07(장별 오버라이드)과 08(i18n)은 미구현이므로 "현재 동작" 대신 "요구사항"을 쓴다. 특히 07은 이 프로젝트의 핵심 설계 문제다: **전역 기본값 + 장별 오버라이드** 구조. 예: 전역은 "F5 / 렌즈 미표시 / Portra 400"인데 3번 사진만 "50mm f/1.8G 표시"로 덮어쓰기. UI는 필름 스트립에서 사진 선택 → 촬영 정보 패널에 "이 장만 다르게" 토글이 뜨는 방향을 제안하되, 구현 시 재검토.

### 1단계 — 공개 가능 최소선 (legacy HTML을 직접 수정)

커밋은 Conventional Commits로 작게 쪼갠다.

1. `fix:` EXIF 회전 — `createImageBitmap(file, { imageOrientation: 'from-image' })` 적용, 미지원 브라우저 폴백
2. `fix:` 캔버스 크기 검사 — 합성 결과가 안전 한계 초과 시 자동 축소 후 토스트로 안내
3. `feat:` JPEG 내보내기 — PNG/JPEG 선택 + 품질(기본 0.92)
4. `docs:` README 작성 — 데모 GIF 자리표시, 한 줄 소개, 만든 이유(동기 스토리), 스크린샷, 사용법, 프라이버시 원칙, 브라우저 지원표(폴더 저장은 Chromium 전용임을 명시), 기술적 결정 요약, 라이선스
5. `ci:` GitHub Pages 배포 워크플로 (정적 파일 그대로)

### 2단계 — 구조 전환

1. Vite + TypeScript 마이그레이션. `vite-plugin-singlefile`로 빌드 결과가 단일 HTML이 되게 유지(다운로드해서 더블클릭으로 여는 사용 방식 보존)
2. **렌더링 엔진 분리** — `src/core/render.ts`: DOM 의존 없는 순수 함수 `render(source, options): Canvas`. 스타일별 렌더러 분리, 타입 정의는 `src/core/types.ts`
3. UI 레이어 분리 — viewport / filmstrip / settings-panel / export
4. 테스트 — 순수 로직(레이아웃 수치 계산, metaLines 조합)은 단위 테스트, 렌더 결과는 픽셀 해시 스냅샷. Vitest 사용
5. 마이그레이션 완료 후 legacy/는 참조용으로만 유지

### 3단계 — 기능 확장 (각각 feature md의 수용 기준을 먼저 갱신하고 구현)

1. 장별 설정 오버라이드 (07 문서 기반)
2. 날짜/장소 필드 (표시 토글에 포함)
3. 프리셋 JSON 내보내기/가져오기
4. 핀치 줌 (모바일)
5. i18n ko/en (08 문서 기반)

### 작업 규칙

- 언어: 문서와 커밋 본문은 한국어, 코드 식별자·커밋 타입은 영어
- 매 단계 완료 시 해당 feature md의 "현재 동작"과 수용 기준 체크 상태를 갱신한다 — 문서와 코드가 어긋난 채 두지 마라
- 새 의존성 추가는 최소화하고, 추가할 때 ARCHITECTURE.md에 이유를 한 줄 남긴다
- 사진 데이터를 외부로 보내는 코드(analytics 포함)는 어떤 이유로도 넣지 않는다
- 불명확한 설계 판단(예: 오버라이드 UI 배치)은 임의로 확정하지 말고 선택지를 제시하고 물어라

### 시작 순서

1. `filmframe.html` 전체를 읽는다
2. 0단계 문서 골격을 만들고 커밋
3. 1단계를 항목별로 커밋하며 진행
4. 1단계가 끝나면 진행 상황을 요약하고 2단계 착수 전에 확인을 받는다
